# Feature-02: Создание бронирования

## Цель функции

Реализовать создание бронирования заезда: после просмотра подробностей доступного слота клиент может заполнить форму с контактными данными, выбрать количество участников и вариант экипировки, отправить запрос через API и получить подтверждение с номером бронирования.

## Реализованное поведение

- Кнопка «Записаться» в деталях слота (только если bookingAllowed=true, статус available, availableKarts > 0)
- Форма бронирования:
  - Дата, время, трасса, маршал, свободные карты, адрес, место сбора (только чтение)
  - Поле «Имя» (обязательно, минимум 2 символа)
  - Поле «Телефон» (обязательно, минимум 10 цифр)
  - Поле «Количество участников» (целое, 1..min(availableKarts, maxParticipants))
  - Радио-выбор экипировки (своя / шлем / шлем+подшлемник / полный комплект)
  - Чекбокс согласия на обработку персональных данных
  - Автоматический пересчёт итоговой стоимости
- Валидация всех полей перед отправкой
- Отправка POST /bookings через API/mock
- Экран подтверждения с номером бронирования (формат APEX-XXXXXX), деталями и стоимостью
- Кнопка «К расписанию» возвращает к списку слотов
- Обработка ошибок:
  - 409 — сообщение о конфликте, кнопка обновления слота
  - 400/422 — сообщение об ошибке данных
  - 404 — слот не найден, кнопка к расписанию
  - 503 — сервис недоступен
- После успешного бронирования availableKarts уменьшается в mock-данных

---

## Временные mock-допущения (требуют подтверждения заказчика)

1. **Один вариант экипировки на всё бронирование.** В текущей mock-реализации клиент выбирает один пакет экипировки (своя, шлем, шлем+подшлемник, полный комплект), который применяется ко всем участникам. Реальная схема может предусматривать индивидуальный выбор экипировки для каждого участника.

2. **basePrice — цена за одного участника.** В mock-реализации `basePrice` слота умножается на количество участников. Реальная ценовая модель может отличаться (фиксированная цена за слот, tiered pricing и т.д.).

3. **Стоимость прокатной экипировки — за каждого участника.** `totalPrice = basePrice × participantsCount + rentalPrice × participantsCount`. Реальная схема расчёта экипировки требует уточнения.

4. **Наличие экипировки не проверяется.** Mock API не проверяет остатки прокатной экипировки. В реальной системе может потребоваться проверка доступного фонда.

## Поведение интерфейса

В карточке и подробностях доступного слота добавь кнопку:

«Записаться»

Кнопка должна быть недоступна, если:

- bookingAllowed равно false;
- слот отменён или закрыт;
- availableKarts равно 0.

После нажатия открывай форму бронирования.

Форма должна показывать:

- дату и время;
- трассу;
- маршала;
- количество свободных картов;
- базовую цену;
- адрес и место сбора.

Поля формы:

- имя клиента;
- телефон;
- количество участников;
- вариант экипировки;
- согласие на обработку персональных данных.

Количество участников:

- целое число;
- минимум 1;
- не больше availableKarts;
- не больше maxParticipants.

Для учебного mock-режима используй временное допущение:

- один вариант экипировки выбирается для всего бронирования;
- basePrice считается ценой за одного участника;
- прокатная экипировка оплачивается за каждого участника;
- totalPrice =
  basePrice × participantsCount +
  rentalPrice × participantsCount.

Это допущение обязательно укажи в документации как требующее
подтверждения заказчика.

## Mock API

В js/api.js добавь метод создания бронирования.

В js/mock-api.js реализуй имитацию:

POST /bookings

Запрос должен содержать:

- slotId;
- participantsCount;
- equipmentSelection;
- customer:
  - name;
  - phone.

Перед созданием повторно проверь:

- слот существует;
- bookingAllowed равно true;
- слот не отменён;
- количество участников допустимо;
- свободных картов достаточно;
- прокатной экипировки достаточно.

При успехе верни объект Booking:

- id;
- bookingNumber формата APEX-XXXXXX;
- slot;
- participantsCount;
- equipmentSelection;
- customer;
- totalPrice;
- status: confirmed;
- canCancel;
- cancellationDeadline;
- cancellationInfo: null;
- createdAt;
- updatedAt.

Для mock-режима допускается хранить созданные бронирования в
LocalStorage под ключом kartingMockBookings.

В документации явно укажи, что это только техническое хранилище
учебного mock API, а не источник истины реальной системы.

После успешного бронирования:

- уменьши availableKarts в mock-данных;
- закрой форму;
- покажи номер бронирования;
- покажи итоговую стоимость;
- обнови список слотов.

## Ошибки

Реализуй обработку:

- 400 — неверные данные;
- 404 — слот не найден;
- 409 — мест или экипировки уже недостаточно;
- 422 — бронирование запрещено;
- 503 — mock API временно недоступен.

При 409:

- покажи понятное сообщение;
- повторно загрузи слот;
- обнови число свободных картов;
- предложи уменьшить число участников или выбрать другой слот.

Ошибки показывай в интерфейсе, без alert.

Не допускай необработанных ошибок в консоли.
