# Баг 3: Отображение устаревшего количества свободных картов

| Поле | Значение |
|------|----------|
| **ID** | BUG-03 |
| **Название** | Количество свободных картов не обновляется после бронирования и отмены |

## Симптом

После создания бронирования в каталоге слотов отображается исходное количество свободных картов, а не уменьшенное. После отмены бронирования карты не возвращаются в отображение. После обновления страницы все изменения количества картов теряются.

### Конкретные проявления

1. Пользователь видит слот с 8 свободными картами.
2. Бронирует 2 места — получает подтверждение.
3. Возвращается в каталог — слот всё ещё показывает 8 картов (должно быть 6).
4. Открывает слота — всё ещё 8 картов.
5. После обновления страницы — slotStates (in-memory Map) сбрасывается, все ajustments теряются.

## Серьёзность

**Высокая** — нарушает FR-05 (availableKarts) и BR-07 (количество участников не может превышать свободные карты). Пользователь может видеть неактуальное количество мест, что приводит к ошибкам 409 при попытке забронировать уже занятый слот.

## Окружение

- Все браузеры
- Все устройства

## Связанные требования

- FR-05: Поле количества участников: максимум = min(availableKarts, maxParticipants)
- BR-07: Количество участников не может превышать свободные карты на момент запроса
- FR-21: LocalStorage используется только для UX — не является источником истины

## Предусловия

- Приложение загружено
- Есть слот с availableKarts > 0
- Нет существующих бронирований на этот слот

## Шаги воспроизведения

1. Открыть каталог слотов — запомнить availableKarts для выбранного слота (например, 8)
2. Открыть слот, нажать «Записаться»
3. Заполнить форму, указать 2 участника
4. Нажать «Забронировать» — получить подтверждение
5. Нажать «К расписанию»
6. Открыть тот же слот — availableKarts = 8 (ожидается 6)
7. Нажать F5 (обновить страницу)
8. Открыть слот — availableKarts = 8

## Ожидаемый результат

После шага 5: availableKarts = исходные − 2 = 6.
После шага 8: availableKarts = исходные − 2 = 6 (данные восстанавливаются из сохранённых бронирований).

## Фактический результат

После шага 5: availableKarts = исходные (запись в slotStates не учитывалась в mockFetchSlots).
После шага 8: availableKarts = исходные (slotStates — in-memory, утерян при перезагрузке).

## Причина

В `mockFetchSlots` и `mockFetchSlot` availableKarts формировался непосредственно из `SLOT_DEFS` без учёта уже созданных бронирований. Параллельно существовал in-memory Map `slotStates`, который обновлялся в `mockCreateBooking` и `mockCancelBooking`, но никогда не читался функциями получения списка/детали слота. Таким образом:

1. `mockFetchSlots` всегда возвращал `def.karts`, игнорируя `slotStates`
2. `slotStates` был in-memory и не сохранялся в localStorage — при обновлении страницы все изменения терялись
3. В `mockCancelBooking` карты возвращались в `slotStates`, но это никак не влияло на отображение

## Исправление

В `js/mock-api.js` реализован новый подход:

1. **Удалён in-memory `slotStates` Map** — источником истины для availableKarts теперь выступают сохранённые бронирования.

2. **Функция `getKarts` переписана** — теперь вычисляет доступные карты на основе `SLOT_DEFS` и confirmed-бронирований из localStorage:
   ```js
   function getKarts(slotId, def) {
     const stored = JSON.parse(localStorage.getItem(LS_KEY) || '[]');
     const bookedCount = stored
       .filter(b => b.slot.id === slotId && b.status === 'confirmed')
       .reduce((sum, b) => sum + b.participantsCount, 0);
     return Math.max(0, def.karts - bookedCount);
   }
   ```

3. **`mockFetchSlots` и `mockFetchSlot`** — после сборки слотов из SLOT_DEFS применяют `getKarts` для каждого слота, чтобы получить актуальное количество свободных картов.

4. **`mockCancelBooking`** — больше не требует ручного возврата картов, так как изменение статуса на `cancelled_by_client` автоматически исключает бронирование из расчёта `getKarts` (фильтр `b.status === 'confirmed'`).

5. **`mockCreateBooking`** — больше не обновляет slotStates после сохранения бронирования; последующие вызовы `getKarts` автоматически учитывают новое бронирование.

### Преимущества подхода

- **Единый источник истины** — availableKarts всегда вычисляется из сохранённых данных
- **Устойчивость к перезагрузке** — после F5 данные восстанавливаются из localStorage
- **Автоматический возврат картов** — при отмене статус меняется, и подтверждённых бронирований становится меньше
- **Нет дублирования состояния** — удалён slotStates, который дублировал данные из localStorage

  - Удалена функция `setKarts`
  - Переписана функция `getKarts` (строки 150–156)

## Ручная проверка

1. Открыть каталог слотов — запомнить availableKarts для слота X
2. Создать бронирование на 2 участников для слота X
3. Вернуться к расписанию — availableKarts для слота X: (исходные − 2)
4. Открыть детали слота X — availableKarts: (исходные − 2)
5. Обновить страницу (F5) — availableKarts: (исходные − 2)
6. Отменить бронирование
7. Вернуться к расписанию — availableKarts: исходные
8. Обновить страницу — availableKarts: исходные
9. Создать бронирование на N участников, где N > availableKarts — ошибка 409
10. Проверить консоль браузера — нет ошибок JavaScript
