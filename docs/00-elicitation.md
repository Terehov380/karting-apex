## 3. Границы скоупа

### Входит в скоуп текущей поставки:

- Адаптивное клиентское веб-приложение (mobile-first)
- Просмотр расписания слотов (7 дней по умолчанию, фильтр периода)
- Детальная информация о слоте: трасса, маршал, свободные карты, стоимость, адрес, место сбора
- Выбор количества участников и экипировки
- Создание бронирования через API
- Просмотр своих бронирований
- Отмена бронирования (с учётом canCancel и cancellationDeadline)
- Отображение статуса «Отменён центром» и причины
- Обработка 409 Conflict
- Mock API для учебной демонстрации (включая уведомления)
- Клиентский API-клиент (js/api.js)

### Не входит в скоуп:

- Интерфейсы администратора и маршала (существуют в инфраструктуре)
- Формирование расписания, управление слотами, трассами, маршалами
- Серверная реализация, БД, транзакционность
- Онлайн-оплата, платёжный шлюз
- Система лояльности / постоянные клиенты
- Оценка маршала после заезда
- Нативные мобильные приложения (Android/iOS)
- PWA, офлайн-режим (не требуется в MVP)
- Email/SMS-уведомления (только mock)
- Учёт погоды (реализуется центром)
- Многоязычность
- Регистрация и авторизация (кроме механизма идентификации, необходимого для API — TBD)

## 4. Противоречия со старой версией документов (до актуализации)

### 4.1. Архитектура

| Старая версия | Новая версия |
|---------------|--------------|
| Полностью клиентское SPA без сервера | API-backed клиент; backend — источник истины |
| Статические тарифы в data.js | Слоты, трассы, маршалы из API |
| LocalStorage — основное хранилище | LocalStorage не является источником истины |
| Гостевой режим без аутентификации | API может возвращать 401; способ идентификации клиента — TBD |

### 4.2. Пользовательский путь

| Старая версия | Новая версия |
|---------------|--------------|
| Каталог тарифов → настройка → корзина → оформление → подтверждение | Список слотов → выбор слота → участники/экипировка → создание брони → подтверждение |
| Корзина с несколькими позициями | Прямое бронирование без корзины |
| Клиентская генерация номера бронирования | Backend возвращает ID бронирования |

### 4.3. Сущности

| Старая версия | Новая версия |
|---------------|--------------|
| Tariff, AvailableTime, CartItem, Customer, Booking | Slot, TrackConfiguration, MarshalSummary, EquipmentOption, Booking, CustomerSummary, CancellationInfo, Notification, APIError |
| Статус: new | Статусы: confirmed, cancelled_by_client, cancelled_by_center, completed |

### 4.4. Функциональность

| Старая версия | Новая версия |
|---------------|--------------|
| Нет отмены бронирования | Отмена с canCancel/cancellationDeadline |
| Нет уведомлений | Mock API уведомлений |
| Нет обработки конфликтов | 409 Conflict |
| Нет empty state | Empty state для расписания |
| Фильтрация по дате не предусмотрена | Фильтр периода по умолчанию + расширенный |

## 5. Открытые вопросы заказчику

Вопросы не были заданы или остались без ответа в исходном брифе и уточнениях:

1. **Идентификация клиента.** Как клиент идентифицируется при запросах к API? По номеру телефона? Через API-ключ? Требуется ли минимальная регистрация?
2. **Привязка бронирований.** Как система понимает, какие бронирования принадлежат данному клиенту (GET /bookings)? По phone + name? По сессионному токену?
3. **Состав экипировки.** Какие именно позиции экипировки предлагаются (шлем, подшлемник, комбинезон?) и какова стоимость каждой?
4. **Формат места сбора.** В каком формате ожидается адрес и место сбора — текстовая строка, координаты, ссылка на карты?
5. **Механизм mock-уведомлений.** Как именно имитируются уведомления в учебной версии — по таймеру, при опросе API, через WebSocket?
6. **406/415/429 ошибки.** Нужны ли эти HTTP-статусы в контракте или достаточно указанных (400, 401, 404, 409, 422, 500, 503)?
7. **Детали 422.** Какие поля могут быть rejected при 422 и в каком формате приходят детали?
