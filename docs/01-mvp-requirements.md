# Karting Drive — требования к MVP

## 1. Общая информация о проекте

| Параметр | Значение |
|----------|----------|
| Название | Karting Drive |
| Тип | Адаптивное клиентское веб-приложение (mobile-first) |
| Предметная область | Бронирование заездов в картинг-центре |
| Стек | HTML, CSS, JavaScript (без фреймворков) |
| Хранение данных | Backend (API) — источник истины; LocalStorage — только для UX (кэш/сессия) |
| Серверная часть | Существующий backend (black-box); клиент — только API-потребитель |
| Реализуемая роль | Только клиент; админка и интерфейс маршала вне скоупа |
| Версия документа | 2.0 (актуализация по брифу) |
| Этап | Аналитика (MVP) |

---

## 2. Описание продукта

**Karting Drive** — адаптивное клиентское веб-приложение для картинг-центра «Апекс», позволяющее пользователю просматривать расписание слотов на 7 дней (с фильтром периода), выбирать слот с информацией о трассе, маршале, свободных картах, стоимости и месте сбора, указывать количество участников и экипировку, создавать бронирование через API, просматривать и отменять свои бронирования с учётом ограничений.

Приложение работает через API существующего backend, который является единственным источником истины. Backend гарантирует отсутствие двойных броней. Клиентское приложение обрабатывает 409 Conflict и другие ошибки API.

---

## 3. Цель проекта

Создать минимально жизнеспособный продукт (MVP), демонстрирующий полный пользовательский путь от просмотра расписания до получения подтверждения бронирования, с использованием API существующего backend, адаптивной mobile-first вёрсткой и обработкой всех ключевых сценариев (включая конфликты и отмены).

---

## 4. Проблема пользователя

Посетитель картинг-центра хочет быстро узнать доступные слоты, их детали (трасса, маршал, цена, свободные карты), выбрать удобное время и забронировать заезд без звонка оператору. Без цифрового инструмента пользователь вынужден писать в Telegram или звонить, что замедляет принятие решения и создаёт риск ошибок при записи.

---

## 5. Ценность продукта

- Актуальное расписание слотов из backend в реальном времени.
- Прозрачное отображение цены, свободных карт, трассы, маршала до бронирования.
- Самостоятельное создание, просмотр и отмена бронирований.
- Уведомление об отмене заезда центром.
- Адаптивный интерфейс для телефона (основной канал использования).

## 7. Границы MVP

**Входит в MVP:**

- Просмотр расписания слотов (по умолчанию 7 дней, с фильтром периода).
- Детальная информация о слоте: трасса, маршал, свободные карты, цена, адрес, место сбора.
- Выбор количества участников и экипировки (своя/прокат).
- Создание бронирования через POST /bookings.
- Просмотр своих бронирований (GET /bookings).
- Отмена бронирования при canCancel=true.
- Отображение статуса «Отменён центром» и причины.
- Обработка ошибок API: 400, 401, 404, 409, 422, 500, 503.
- Empty state при отсутствии слотов.
- Mock API для учебной демонстрации.

**Не входит в MVP:**

- Интерфейсы администратора и маршала.
- Формирование/редактирование расписания, слотов, трасс, маршалов.
- Серверная реализация, БД, транзакционность (backend существует).
- Онлайн-оплата, платёжный шлюз.
- Система лояльности / постоянные клиенты.
- Оценка маршала после заезда.
- PWA / офлайн-режим.
- Email/SMS-уведомления (только mock в учебной версии).
- Учёт погоды (центр уведомляет через отмену слота).
- Многоязычность.
- Регистрация и полноценная авторизация.
- Определение механизма идентификации клиента (TBD, ожидает решения заказчика).

## 8. User Stories

### US-01. Просмотр расписания на 7 дней

**Как** клиент,
**я хочу** видеть доступные слоты на ближайшие 7 дней по умолчанию,
**чтобы** выбрать удобное время для заезда.

### US-02. Фильтр периода

**Как** клиент,
**я хочу** изменить период отображения слотов (больше или меньше 7 дней),
**чтобы** планировать заезд на произвольную дату.

### US-03. Empty state при отсутствии слотов

**Как** клиент,
**я хочу** видеть понятное сообщение, если на выбранный период нет слотов,
**чтобы** понимать, что расписание пустое, а не из-за ошибки.

### US-04. Детальная информация о слоте

**Как** клиент,
**я хочу** видеть стоимость, свободные карты, трассу, маршала, адрес и место сбора выбранного слота,
**чтобы** принять решение о бронировании.

### US-05. Выбор количества участников

**Как** клиент,
**я хочу** указать количество участников для выбранного слота,
**чтобы** забронировать на всю группу.

### US-06. Выбор экипировки

**Как** клиент,
**я хочу** выбрать, беру ли я экипировку напрокат или приезжаю со своей,
**чтобы** получить полную стоимость.

### US-07. Создание бронирования

**Как** клиент,
**я хочу** создать бронирование после выбора слота, участников и экипировки,
**чтобы** подтвердить заезд.

### US-08. Просмотр своих бронирований

**Как** клиент,
**я хочу** видеть список всех моих бронирований с деталями и статусом,
**чтобы** контролировать свои записи.

### US-09. Детальный просмотр бронирования

**Как** клиент,
**я хочу** открыть конкретное бронирование и увидеть все его детали,
**чтобы** проверить информацию о заезде.

### US-10. Отмена бронирования

**Как** клиент,
**я хочу** отменить бронирование (если разрешено),
**чтобы** освободить слот, если планы изменились.

### US-11. Индикация невозможности отмены

**Как** клиент,
**я хочу** видеть, что отмена недоступна (canCancel=false, истёк дедлайн),
**чтобы** не пытаться отменить в неподходящий момент.

### US-12. Уведомление об отмене центром

**Как** клиент,
**я хочу** получить уведомление, если центр отменил мой заезд, с указанием причины,
**чтобы** не приехать зря.

### US-13. Обработка 409 Conflict

**Как** клиент,
**я хочу** получить понятное сообщение, если слот уже занят,
**чтобы** выбрать другой слот.

## 9. Критерии приёмки (Acceptance Criteria)

### US-01. Просмотр расписания на 7 дней

- [ ] При загрузке страницы отображаются слоты на ближайшие 7 дней от текущей даты.
- [ ] Слоты сгруппированы по дням.
- [ ] Для каждого слота видны: время старта, трасса, маршал, цена, свободные карты.
- [ ] Данные загружаются через API (или mock-api).

### US-02. Фильтр периода

- [ ] Доступен элемент выбора периода (dateFrom / dateTo).
- [ ] При изменении периода слоты перезагружаются.
- [ ] Фильтр позволяет выбрать произвольный диапазон не менее 1 дня.
- [ ] Кнопка «Сбросить» возвращает к 7 дням по умолчанию.

### US-03. Empty state

- [ ] Если на выбранный период нет доступных слотов, отображается сообщение «Пока нет доступных заездов».
- [ ] Empty state не показывается во время загрузки.
- [ ] Empty state не показывается при ошибке сети (отдельное сообщение об ошибке).

### US-04. Детальная информация о слоте

- [ ] При выборе слота отображаются: трек (название, описание), маршал (имя), свободные карты, цена за участника, полная стоимость (рассчитанная), адрес центра, место сбора.
- [ ] Информация загружается через GET /slots/{slotId} (или из кэша списка).

### US-05. Выбор количества участников

- [ ] Доступен элемент ввода количества участников (number, stepper).
- [ ] Минимум — 1, максимум — maxParticipants слота (но не более availableKarts).
- [ ] При изменении пересчитывается итоговая стоимость.

### US-06. Выбор экипировки

- [ ] Отображается список доступной экипировки с ценой за единицу.
- [ ] Можно выбрать количество единиц каждого типа (не более participants).
- [ ] Можно не выбирать экипировку (клиент со своей).
- [ ] Стоимость экипировки добавляется к итоговой цене.

### US-07. Создание бронирования

- [ ] После заполнения всех полей (имя, телефон, участники, экипировка) доступна кнопка «Забронировать».
- [ ] При нажатии выполняется POST /bookings.
- [ ] При успехе (201) отображается экран подтверждения с ID бронирования, деталями и статусом.
- [ ] При 409 отображается сообщение о конфликте; клиент может выбрать другой слот.
- [ ] При 422 отображаются ошибки валидации полей.
- [ ] Имя и телефон обязательны (мин. 2 символа имя, ≥ 10 цифр телефон).

### US-08. Просмотр своих бронирований

- [ ] Отображается список бронирований с: слот, дата/время, трасса, маршал, участники, статус, цена.
- [ ] Статусы отображаются на русском: «Подтверждено», «Отменено», «Отменён центром», «Завершено».
- [ ] Данные загружаются через GET /bookings.

### US-09. Детальный просмотр бронирования

- [ ] При выборе бронирования из списка открывается детальная информация.
- [ ] Отображаются все поля бронирования, включая cancellationDeadline и canCancel.
- [ ] Если статус cancelled_by_center — отображается причина отмены.

### US-10. Отмена бронирования

- [ ] Кнопка отмены активна только при canCancel=true.
- [ ] При нажатии выполняется POST /bookings/{id}/cancel.
- [ ] После успешной отмены статус обновляется на cancelled_by_client.
- [ ] Кнопка отмены исчезает или становится неактивной.

### US-11. Индикация невозможности отмены

- [ ] Если canCancel=false, кнопка отмены неактивна или скрыта.
- [ ] Отображается причина: «Срок отмены истёк» или «Отмена недоступна».
- [ ] Отображается cancellationDeadline, если он ещё актуален.

### US-12. Уведомление об отмене центром

- [ ] В интерфейсе отображаются уведомления (в учебной версии — из mock-api).
- [ ] Уведомление содержит: тип, сообщение, дату, связанное бронирование.
- [ ] Непрочитанные уведомления визуально выделены.
- [ ] При клике на уведомление открывается соответствующее бронирование.

### US-13. Обработка 409 Conflict

- [ ] При 409 от POST /bookings отображается понятное сообщение: «Выбранный слот больше не имеет свободных мест».
- [ ] Клиент остаётся на экране выбора слота и может выбрать другой.
- [ ] При 409 от POST /bookings/{id}/cancel отображается причина отказа в отмене.

## 10. Основные пользовательские сценарии

### Сценарий 1. Успешное бронирование

1. Клиент открывает приложение.
2. Видит список доступных слотов на 7 дней.
3. Использует фильтр периода, чтобы уточнить дату (опционально).
4. Выбирает подходящий слот.
5. Видит детали: трасса «Короткая», маршал «Алексей С.», свободно 8 картов, цена 1000 ₽/чел, адрес и место сбора.
6. Выбирает количество участников: 2.
7. Выбирает экипировку: 2 шлема (200 ₽/шт).
8. Видит итоговую стоимость: 2 × 1000 + 2 × 200 = 2400 ₽.
9. Вводит имя «Иван Петров» и телефон «+7 999 123-45-67».
10. Нажимает «Забронировать».
11. Получает подтверждение с ID бронирования bk-42 и статусом «Подтверждено».

### Сценарий 2. Конфликт при бронировании (409)

1. Клиент выбирает слот, заполняет данные, нажимает «Забронировать».
2. Backend возвращает 409 — слот занят.
3. Приложение показывает сообщение: «Выбранный слот больше не имеет свободных мест».
4. Клиент возвращается к списку слотов и выбирает другой.
5. Повторяет бронирование успешно.

### Сценарий 3. Отмена бронирования

1. Клиент открывает «Мои бронирования».
2. Видит бронь со статусом «Подтверждено» и активной кнопкой «Отменить».
3. Нажимает «Отменить».
4. Подтверждает отмену.
5. Статус меняется на «Отменено», кнопка исчезает.

### Сценарий 4. Попытка отмены после дедлайна

1. Клиент открывает бронирование.
2. canCancel=false, cancellationDeadline в прошлом.
3. Кнопка отмены неактивна, отображается «Срок отмены истёк».

### Сценарий 5. Отмена заезда центром

1. Центр отменяет слот (вне клиентского приложения).
2. Клиент получает уведомление: «Ваш заезд 5 июля в 10:00 отменён центром. Причина: сильный дождь».
3. Клиент открывает бронирование — статус «Отменён центром», причина указана.

### Сценарий 6. Пустое расписание

1. Клиент открывает приложение.
2. На выбранный период нет слотов.
3. Отображается empty state: «Пока нет доступных заездов».
4. Клиент меняет период или проверяет позже.

## 11. Альтернативные сценарии и обработка ошибок

| Ситуация | Поведение системы |
|----------|-------------------|
| Нет слотов на выбранный период | Empty state: «Пока нет доступных заездов» |
| Ошибка сети / 503 | Сообщение: «Сервис временно недоступен. Попробуйте позже» |
| 401 при запросе | Сообщение: «Ошибка доступа. Перезагрузите страницу или обратитесь в центр» (детали — после определения механизма идентификации клиента) |
| 500 при запросе | Сообщение: «Произошла ошибка. Попробуйте позже» |
| 409 при POST /bookings | Сообщение: «Выбранный слот больше не имеет свободных мест» |
| 409 при POST /bookings/{id}/cancel | Сообщение: «Отмена невозможна. Причина: {message из API}» |
| 422 при POST /bookings | Отображение ошибок валидации по полям (имя, телефон, участники) |
| 404 при GET /slots/{slotId} | Сообщение: «Слот не найден. Вернитесь к расписанию» |
| 404 при GET /bookings/{bookingId} | Сообщение: «Бронирование не найдено» |
| Поле имя пустое | «Введите имя» |
| Поле телефон невалидное | «Введите корректный номер телефона» |
| Участников больше свободных карт | «Свободных картов: N. Уменьшите количество участников» |
| Участников больше maxParticipants | «Максимум N участников для данного слота» |
| Двойной клик «Забронировать» | Блокировка кнопки после первого нажатия (prevent double submit) |

## 12. Функциональные требования

| ID | Требование |
|----|------------|
| **FR-01** | Приложение загружает список слотов через GET /slots при старте (dateFrom=today, dateTo=today+7) |
| **FR-02** | Приложение предоставляет фильтр периода (dateFrom, dateTo) с возможностью сброса |
| **FR-03** | Если API возвращает пустой массив слотов, отображается empty state |
| **FR-04** | При выборе слота отображаются: track.name, track.description, marshal.name, availableKarts, pricePerPerson, location.address, location.meetingPoint |
| **FR-05** | Поле количества участников: минимум 1, максимум = min(availableKarts, maxParticipants) слота |
| **FR-06** | Стоимость пересчитывается при изменении количества участников и экипировки |
| **FR-07** | Выбор экипировки: список EquipmentOption с количеством (0 или 1+), не более participants каждого типа |
| **FR-08** | POST /bookings отправляет slotId, participants, equipment[], customerName, customerPhone |
| **FR-09** | При успешном ответе 201 отображается экран подтверждения с ID, статусом, деталями |
| **FR-10** | При 409 отображается сообщение о конфликте; клиент может выбрать другой слот |
| **FR-11** | При 422 отображаются ошибки валидации по полям |
| **FR-12** | GET /bookings возвращает список бронирований клиента |
| **FR-13** | Каждое бронирование отображает: id, startTime, trackName, marshalName, participants, totalPrice, status, canCancel, cancellationDeadline |
| **FR-14** | Статусы отображаются на русском: confirmed → «Подтверждено», cancelled_by_client → «Отменено», cancelled_by_center → «Отменён центром», completed → «Завершено» |
| **FR-15** | Кнопка отмены активна только при canCancel=true |
| **FR-16** | POST /bookings/{bookingId}/cancel отправляет reason (опционально) |
| **FR-17** | При cancelled_by_center отображаются cancellationInfo.cancelledBy, cancellationInfo.reason |
| **FR-18** | Повторная запись на отменённый центром слот запрещена (слот недоступен) |
| **FR-19** | API-клиент обрабатывает HTTP-статусы: 400, 401, 404, 409, 422, 500, 503 |
| **FR-20** | Mock API реализует все endpoints с тестовыми данными для демонстрации |
| **FR-21** | LocalStorage используется только для UX (предзаполнение имени/телефона) — не является источником истины |
| **FR-22** | Уведомления загружаются через mock API и отображаются в интерфейсе |
| **FR-23** | Навигация между экранами (расписание, бронирование, мои брони, детали) без перезагрузки страницы |

| **NFR-03** | Время до первого отображения контента — не более 3 секунд (локально или static-сервер) |
| **NFR-04** | Не используются фреймворки и библиотеки (React, Vue, jQuery и т.д.) |
| **NFR-05** | Код разделён на логические модули/файлы (HTML, CSS, JS) |
| **NFR-06** | Интерфейс на русском языке |
| **NFR-07** | Все пользовательские сообщения об ошибках понятны и на русском языке |
| **NFR-08** | Приложение не отправляет данные на сторонние серверы (кроме API картинг-центра) |
| **NFR-09** | Кнопка отправки формы блокируется при повторном нажатии (prevent double submit) |
| **NFR-10** | Сетевая ошибка не приводит к «белому экрану» — отображается сообщение |
| **NFR-11** | HTML-разметка семантична (header, main, section, nav) |
| **NFR-12** | Интерактивные элементы доступны с клавиатуры (focus, Enter/Space) |
| **NFR-13** | При ошибке API данные на экране не сбрасываются (кроме случаев, когда это неизбежно) |

## 14. Бизнес-правила

| ID | Правило |
|----|---------|
| **BR-01** | Backend является единственным источником истины; клиент не кэширует данные надолго |
| **BR-02** | Отсутствие двойных броней гарантирует backend (атомарная проверка) |
| **BR-03** | Отмена бронирования возможна только при canCancel=true |
| **BR-04** | Повторная запись на отменённый центром слот запрещена |
| **BR-05** | Бронирование считается успешным без оплаты; оплата — на месте |
| **BR-06** | Контактные данные (имя, телефон) обязательны для бронирования |
| **BR-07** | Количество участников не может превышать свободные карты на момент запроса |
| **BR-08** | Экипировка не обязательна (клиент может приехать со своей) |
| **BR-09** | Клиент не может изменить бронирование — только отменить и создать новое |
| **BR-10** | При cancelled_by_center бронь не удаляется, а меняет статус (R-008) |

## 15. Критерии готовности MVP (Definition of Done)

MVP считается готовым, когда выполнены **все** пункты:

- [ ] Реализованы все user stories US-01 — US-13.
- [ ] Выполнены все функциональные требования FR-01 — FR-23.
- [ ] Выполнены все нефункциональные требования NFR-01 — NFR-13.
- [ ] Соблюдены все бизнес-правила BR-01 — BR-10.
- [ ] Все основные сценарии из раздела 10 проходят ручную проверку без ошибок.
- [ ] Все альтернативные сценарии из раздела 11 обрабатываются корректно.
- [ ] API-клиент корректно обрабатывает все HTTP-статусы (400, 401, 404, 409, 422, 500, 503).
- [ ] Mock API предоставляет все endpoints с тестовыми данными.
- [ ] Приложение отображается корректно на экранах от 320px до 1920px.
- [ ] Отсутствуют критические баги, блокирующие основной пользовательский путь.
- [ ] Код находится в Git-репозитории.
