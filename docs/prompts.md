# Промпты, использованные при разработке

## Промпт 1. Анализ требований к MVP

Ты работаешь в пустом проекте веб-приложения картинг-центра «Karting Drive».

Выполни первый этап проекта — аналитику. Не ограничивайся выводом текста в чат: самостоятельно создай необходимые папки и Markdown-файлы в текущем проекте и заполни их.

## Контекст проекта

Karting Drive — учебное клиентское веб-приложение для картинг-центра.

Пользователь должен иметь возможность:

* просматривать тарифы;
* изучать описание и ограничения тарифа;
* выбирать дату и время заезда;
* указывать количество участников;
* видеть рассчитанную стоимость;
* добавлять заезд в корзину;
* редактировать и удалять позиции;
* вводить имя и телефон;
* оформлять бронирование;
* получать номер бронирования.

## Технологии и ограничения

* HTML;
* CSS;
* JavaScript без фреймворков;
* LocalStorage;
* Git;
* Markdown;
* без серверной части;
* без базы данных;
* без онлайн-оплаты;
* без регистрации и авторизации;
* без личного кабинета;
* без административной панели.

## Что необходимо сделать

### 1. Создай структуру

Создай папку:

`docs`

Создай в ней файлы:

* `docs/01-mvp-requirements.md`;
* `docs/prompts.md`.

Если папка или файлы уже существуют, не удаляй их без необходимости.

### 2. Заполни `docs/01-mvp-requirements.md`

Документ должен содержать:

1. общую информацию о проекте;
2. описание продукта;
3. цель проекта;
4. проблему пользователя;
5. ценность продукта;
6. целевую аудиторию;
7. границы MVP;
8. функции, входящие в MVP;
9. функции, не входящие в MVP;
10. роли пользователей;
11. основные сущности предметной области;
12. user stories;
13. проверяемые критерии приёмки для каждой user story;
14. основные пользовательские сценарии;
15. альтернативные сценарии и обработку ошибок;
16. функциональные требования с идентификаторами `FR-01`, `FR-02` и далее;
17. нефункциональные требования с идентификаторами `NFR-01`, `NFR-02` и далее;
18. бизнес-правила;
19. критерии готовности MVP.

User stories оформи по формату:

**Как** [роль],
**я хочу** [действие],
**чтобы** [ценность].

Обязательно опиши:

* просмотр тарифов;
* просмотр ограничений;
* выбор даты;
* запрет прошедшей даты;
* выбор времени;
* выбор количества участников;
* проверку ограничений тарифа;
* расчёт стоимости;
* добавление в корзину;
* просмотр корзины;
* изменение количества участников;
* удаление позиции;
* сохранение корзины в LocalStorage;
* восстановление корзины после обновления страницы;
* ввод имени и телефона;
* запрет оформления пустой корзины;
* создание бронирования;
* генерацию номера бронирования;
* очистку корзины после оформления.

### 3. Заполни `docs/prompts.md`

Сохрани в этом файле полный текст текущего промпта.

Добавь раздел:

`# Промпты, использованные при разработке`

Текущий промпт оформи как:

`## Промпт 1. Анализ требований к MVP`

### 4. Проверь результат

После создания документов проверь:

* отсутствие противоречий и повторов;
* соответствие ограничениям MVP;
* наличие критериев приёмки у всех user stories;
* возможность ручной проверки требований;
* корректность Markdown-разметки;
* отсутствие требований к серверу, оплате и авторизации.

Не создавай пока архитектуру, схему данных, HTML, CSS и JavaScript. На этом этапе нужна только аналитика.

### 5. Выведи отчёт

После завершения кратко укажи:

* какие файлы созданы или изменены;
* какие разделы добавлены;
* были ли обнаружены противоречия;
* что необходимо делать на следующем этапе.

Если Git уже инициализирован, подготовь изменения к commit, но не выполняй commit без отдельной команды.
