# Промпты, использованные при разработке

## Промпт 1. Анализ требований к MVP

Ты работаешь в пустом проекте веб-приложения картинг-центра «Karting Drive».

Выполни первый этап проекта — аналитику. Не ограничивайся выводом текста в чат: самостоятельно создай необходимые папки и Markdown-файлы в текущем проекте и заполни их.

## Контекст проекта

Karting Drive — учебное клиентское веб-приложение для картинг-центра.

Пользователь должен иметь возможность:

* просматривать тарифы;
* изучать описание и ограничения тарифа;
* выбирать дату и время заезда;
* указывать количество участников;
* видеть рассчитанную стоимость;
* добавлять заезд в корзину;
* редактировать и удалять позиции;
* вводить имя и телефон;
* оформлять бронирование;
* получать номер бронирования.

## Технологии и ограничения

* HTML;
* CSS;
* JavaScript без фреймворков;
* LocalStorage;
* Git;
* Markdown;
* без серверной части;
* без базы данных;
* без онлайн-оплаты;
* без регистрации и авторизации;
* без личного кабинета;
* без административной панели.

## Поведение интерфейса

В карточке и подробностях доступного слота добавь кнопку:

«Записаться»

Кнопка должна быть недоступна, если:

- bookingAllowed равно false;
- слот отменён или закрыт;
- availableKarts равно 0.

После нажатия открывай форму бронирования.

Форма должна показывать:

- дату и время;
- трассу;
- маршала;
- количество свободных картов;
- базовую цену;
- адрес и место сбора.

Поля формы:

- имя клиента;
- телефон;
- количество участников;
- вариант экипировки;
- согласие на обработку персональных данных.

Количество участников:

- целое число;
- минимум 1;
- не больше availableKarts;
- не больше maxParticipants.

Для учебного mock-режима используй временное допущение:

- один вариант экипировки выбирается для всего бронирования;
- basePrice считается ценой за одного участника;
- прокатная экипировка оплачивается за каждого участника;
- totalPrice =
  basePrice × participantsCount +
  rentalPrice × participantsCount.

Это допущение обязательно укажи в документации как требующее
подтверждения заказчика.

## Mock API

В js/api.js добавь метод создания бронирования.

В js/mock-api.js реализуй имитацию:

POST /bookings

Запрос должен содержать:

- slotId;
- participantsCount;
- equipmentSelection;
- customer:
  - name;
  - phone.

Перед созданием повторно проверь:

- слот существует;
- bookingAllowed равно true;
- слот не отменён;
- количество участников допустимо;
- свободных картов достаточно;
- прокатной экипировки достаточно.

При успехе верни объект Booking:

- id;
- bookingNumber формата APEX-XXXXXX;
- slot;
- participantsCount;
- equipmentSelection;
- customer;
- totalPrice;
- status: confirmed;
- canCancel;
- cancellationDeadline;
- cancellationInfo: null;
- createdAt;
- updatedAt.

Для mock-режима допускается хранить созданные бронирования в
LocalStorage под ключом kartingMockBookings.

В документации явно укажи, что это только техническое хранилище
учебного mock API, а не источник истины реальной системы.

После успешного бронирования:

- уменьши availableKarts в mock-данных;
- закрой форму;
- покажи номер бронирования;
- покажи итоговую стоимость;
- обнови список слотов.

## Ошибки

Реализуй обработку:

- 400 — неверные данные;
- 404 — слот не найден;
- 409 — мест или экипировки уже недостаточно;
- 422 — бронирование запрещено;
- 503 — mock API временно недоступен.

При 409:

- покажи понятное сообщение;
- повторно загрузи слот;
- обнови число свободных картов;
- предложи уменьшить число участников или выбрать другой слот.

Ошибки показывай в интерфейсе, без alert.

Не допускай необработанных ошибок в консоли.

## Интерфейс

Добавь в навигацию раздел:

«Мои бронирования»

При открытии раздела загружай бронирования через mock API.

Для каждого бронирования отображай:

- номер бронирования;
- дату и время заезда;
- конфигурацию трассы;
- маршала;
- количество участников;
- выбранную экипировку;
- итоговую стоимость;
- текущий статус;
- дату создания;

- причину отмены, если она есть.

Поддерживаемые статусы:

- confirmed — «Подтверждено»;
- cancelled_by_client — «Отменено клиентом»;
- cancelled_by_center — «Отменено центром»;
- completed — «Завершено».

Статусы должны визуально отличаться.

Если бронирований нет, показывай empty state:

«У вас пока нет бронирований».

## Отмена бронирования

Кнопка «Отменить бронирование» отображается только если:

- status равно confirmed;
- canCancel равно true;
- cancellationDeadline ещё не наступил.

Перед отменой покажи встроенное подтверждение в интерфейсе,
не используй alert и confirm.

После подтверждения:

- отправь запрос отмены;
- измени статус на cancelled_by_client;
- установи canCancel: false;
- создай cancellationInfo:
  - reason: «Отменено клиентом»;
  - cancelledAt;
  - cancelledBy: client;
- обнови updatedAt;
- верни занятые карты в соответствующий слот mock API;
- обнови список бронирований и каталог слотов.

Если отмена запрещена, покажи причину.

## Отмена центром

Для демонстрации добавь в mock API одно тестовое бронирование
со статусом cancelled_by_center или механизм имитации такого статуса.

Для cancelled_by_center:

- бронирование не удаляется;
- отображается причина отмены;
- отображается дата отмены;
- кнопка отмены отсутствует;
- повторное бронирование отменённого слота запрещено.

Не создавай интерфейс администратора и не добавляй кнопку
«Отменить центром».
